# 智慧校园平台 - AI辅助编程记录

## 1. 编程工作量分布与协作模式概述
本项目在开发过程中，将 AI 大语言模型（主要使用 DeepSeek、Claude 等基座模型）深度融入了研发工作流。我们不仅将其作为代码补全工具，更将其定位为“高级技术顾问”与“代码审查员”。系统的主体架构、业务逻辑边界、前后端数据链路及核心UI组件均由本人自主设计与手写，AI 主要承担特定领域的技术难点攻坚与高门槛算法的辅助实现。

### 1.1 自主编写模块（个人代码贡献占比约 75%）
系统的主体骨架和常规业务逻辑均为手工敲击代码，确保系统核心命脉完全可控：
- **全栈工程脚手架**：基于 React 18 + TypeScript + Vite 的前端工程化配置，以及基于 FastAPI 的后端 ASGI 应用挂载。
- **业务UI与交互流**：`ChatPage.tsx`, `LoginPage.tsx`, `RegisterPage.tsx` 的所有 Tailwind CSS 响应式布局与交互状态管理。
- **身份安全与鉴权网关**：基于 bcrypt 的密码哈希加密，以及无状态 JWT (JSON Web Token) 的签发、前端 Axios 拦截器注入、路由守卫拦截机制。
- **持久化层设计**：SQLAlchemy 异步数据库模型（ORM）架构，基于 `database.py` 的会话池管理，以及校园知识库、对话记录的标准 CRUD 业务接口。
- **系统集成与测试**：各模块间的通信组装、CORS跨域配置、`config.py` 环境解耦，以及最终的上线联调。

### 1.2 AI辅助开发模块（AI辅助代码占比约 25%）
AI 在以下涉及高阶算法、数学计算及生僻底层 API 调用的“硬核”模块中提供了不可替代的支持：
- **MAS (多智能体) 调度算法优化**：本人搭建了 `agent_orchestrator.py` 的整体框架，AI 辅助补充了基于权重的任务路由树算法，并协助实现了高并发场景下的 Redis 哈希拦截缓存锁，防止请求击穿。
- **RAG 向量检索底层逻辑**：针对 FAISS 引擎的底层 API 操作，AI 辅助生成了 Embedding 降维映射及 `faiss.normalize_L2` 归一化配合内积（Inner Product）计算余弦相似度的核心代码片段。
- **大模型幻觉熔断防御**：在 `ai_service.py` 接入外部大模型 API 时，AI 辅助构建了极具鲁棒性的 Prompt Engineering 模板，并协助编写了网络抖动重试链路及异常熔断回退代码。
- **前端流式渲染性能调优**：在解析 SSE (Server-Sent Events) 的流式 Markdown 响应时，AI 辅助解决了 React 状态频繁重绘导致的卡顿，提供了一套基于队列缓冲的渲染调优方案。

## 2. 关键 AI 交互日志与 Prompt 策略记录

我们在协作中总结出了一套高效的 Prompt (提示词) 沟通法则：**“提供上下文 + 明确输入输出约束 + 限制修改范围”**。以下为典型交互切片：

### 交互记录一：攻克前端 SSE 流式响应渲染卡顿
- **我的输入 (User)**：
  > [前置代码片段...] 
  > “我在开发前端对话框 `ChatMessage.tsx` 时，使用 SSE 接收后端的流式文本。目前把每次收到的 chunk 直接通过 `setMessages` 更新到状态里，但如果后端吐字极快，页面会疯狂重新渲染，输入框都卡死了。请不要修改我的样式代码，只针对状态更新机制给出一个性能优化方案。是否可以用 requestAnimationFrame 或缓冲队列？”
- **AI 辅助输出**：
  > “频繁触发 React 的 setState 确实会导致 Diff 算法耗尽主线程。你的思路很对，可以采用缓冲队列（Buffer Queue）机制。
  > 建议：在组件外或使用 `useRef` 维护一个字符串 buffer，累加收到的 chunk；然后使用节流（Throttle）或 `requestAnimationFrame` 定期将 buffer 同步给 React 状态。
  > 代码实现示例：...”
- **协作结果**：我将 AI 提供的 buffer 定期刷新思想剥离出来，手动融合进了现有的自定义 Hook 中，完美实现了丝滑的打字机输出体验。

### 交互记录二：设计 MAS 路由防并发拦截机制
- **我的输入 (User)**：
  > “后端的 Agent 调度网关现在有个问题：如果用户因为网络慢，连续疯狂点击‘发送’按钮，会瞬间发起多个相同的复杂请求，压垮数据库和大模型。我想用 Redis 做一个基于用户请求内容的防重复锁，你给我写一个基于 `aioredis` 的异步拦截装饰器或拦截函数。”
- **AI 辅助输出**：
  > “这是一个非常典型的接口防抖与幂等性问题。可以使用 MD5 将 (用户ID + 消息内容) 映射为唯一的 Redis Key，设置 5 秒过期时间。
  > Python 异步实现如下：
  > [异步 Redis 锁代码片段...]”
- **协作结果**：我调整了 AI 给出的装饰器方案，将其改造为 `AgentOrchestrator` 类内部的 `_route_request` 方法级拦截，并补全了相关的错误日志输出。

### 交互记录三：FAISS 向量检索召回率的数学调优
- **我的输入 (User)**：
  > “目前的 RAG 服务中，`rag_service.py` 查出来的相似度结果不太准确，有些高度相关的内容排在很后面。我目前使用的是 `faiss.IndexFlatL2`，怎样能实现更精准的余弦相似度检索？”
- **AI 辅助输出**：
  > “L2 距离与余弦相似度在向量未归一化时表现不一致。如果你的 Embedding 模型推荐使用余弦相似度，建议改用内积索引 `faiss.IndexFlatIP`，并且**必须在存入和查询前对向量进行 L2 归一化**。
  > 核心代码修改如下：
  > `faiss.normalize_L2(embedding_matrix)` 
  > `index = faiss.IndexFlatIP(dimension)`”
- **协作结果**：直接采纳。这个属于纯底层的数学算法知识盲区，AI 的指导帮我节省了至少一天的翻阅文档时间。

## 3. 效能提升统计与方法论总结

### 3.1 研发效能数据预估
- **代码查错提效**：遇到如 `ResourceClosedError` 或前端闭包陷阱等隐蔽 Bug 时，传统 Google/StackOverflow 搜索通常需要 1-2 小时，利用 AI 直接分析堆栈日志，排查时间缩短至 15 分钟内。
- **底层方法调用**：对于不熟悉的库（如 FAISS、aiosqlite 锁机制），免去了从头阅读冗长英文官方文档的成本。
- **整体提效**：保守估计，在核心难点攻坚上，AI 辅助将项目总耗时缩减了约 30%。

### 3.2 最终反思
在此次项目实战中，我深刻认识到：**AI 是极好的“战术执行者”，但绝对不能成为“战略决策者”**。
系统的数据表应当如何关联、前端组件如何拆分、API 的路由规范应当如何设计——这些关乎系统全局稳定性、可维护性与一致性的宏观架构，如果交由 AI 自由发挥，必定会生成一堆看似能跑但结构混乱的“屎山代码”。
因此，最佳的协作方式是：**“开发者绘制系统蓝图与骨架，AI 负责在限定的局部函数内填补砖瓦”**。只有自己亲手掌控了数据流与架构底座，才能在 Bug 出现时游刃有余，让 AI 真正成为研发利器而非失控的隐患。
