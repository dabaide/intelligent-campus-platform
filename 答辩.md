## 1. 项目概述

### 1.1 项目背景

传统校园服务面临三大痛点：

| 痛点 | 具体表现 |
|------|---------|
| **信息碎片化** | 制度文件分散在各部门官网、公告栏、手册中，学生查找困难 |
| **咨询效率低** | 教务/学工窗口排队耗时，人工回答一致性差、易出错 |
| **服务时间受限** | 办公室只在工作日开放，学生课后/周末无法获取帮助 |

### 1.2 解决方案

本项目构建了一个 **基於多智能体系统（MAS）+ 检索增强生成（RAG）** 的校园知识服务平台，核心特性：

```
用户提问 → [NavigationAgent 意图识别]
                │
        ┌───────┼───────┬──────────┐
        ▼       ▼       ▼          ▼
    Academic  Life   Admin     General
    (学术)   (生活)  (行政)    (通用)
        │       │       │
        └───────┴───────┘
                │
                ▼
        [QualityAgent 质量验证]
                │
                ▼
          返回精准回复
```

### 1.3 五大智能体一览

| 智能体 | 核心职责 | 典型场景 |
|--------|---------|---------|
| **NavigationAgent** | 意图识别与路由分发 | 用户说"你好"→ 问候；用户问"奖学金"→ 路由到 Admin |
| **AcademicAgent** | 学朮事务 | 选课、成绩、学分、培养方案、考试 |
| **LifeServiceAgent** | 生活服务 | 食堂、图书馆、宿舍、校车、校园卡 |
| **AdminAgent** | 🎯 **行政事务** | 奖学金、请假、报销、证明、学籍、毕业 |
| **QualityAgent** | 质量验证 | 防幻觉检测、事实核查、交叉验证 |

```
     [用户提问]
         │
         ▼

         │
    ┌────┴────┐
    │ 命中FAQ? │

    │         │
    │         ▼
    │    QualityAgent
    │    .verify_response()
    │         │
    │    ┌────┴────┐
    │    │ 幻觉检测 │
    │    │ 事实核查 │
    │    │ 来源验证 │

    │         │
    └────┬────┘
         │
         ▼
    [最终回复用户]
```

**三层防控**：
1. **FAQ 直接返回**：命中高置信度 FAQ 时直接返回标准答案，跳过 AI 生成 → 零幻觉
2. **RAG 上下文约束**：通过 Prompt Engineering 强制大模型仅依据知识库内容回答
3. **QualityAgent 后置验证**：检测不支持的声明、虚假电话号码、日期等具体幻觉信号

### 3.2 请求处理全流程

```
1. 用户发送消息
   POST /api/v1/chat/{conversation_id}/message
   Authorization: Bearer <JWT>

2. JWT 验证中间件
   └─ 解析 token → 提取 user_id, role, department

3. AgentOrchestrator.process_message()
   ├─ ① MD5(query + user_id) → Redis 缓存检查
   │    命中 → 直接返回 (防重复请求)
   │
   ├─ ② NavigationAgent.analyze_intent()
   │    ├─ 问候检测 → 直接回复
   │    ├─ 关键词正则匹配 → 强制路由
   │    └─ DeepSeek AI 意图分析 → 智能路由
   │
   ├─ ③ 路由到专业智能体
   │    admin → AdminAgent.process_request()
   │
   ├─ ④ RAG 增强 + DeepSeek 生成
   │    AI 仅在知识库不足时介入
   │
   ├─ ⑤ QualityAgent.verify_response()
   │    质量评分 + 幻觉检测
   │
   └─ ⑥ 记录 AgentInteraction → 数据库
        记录响应 → Redis 缓存 (30分钟TTL)
```

### 4.1 行政智能体核心类 (`backend/agents/admin_agent.py`)

```python
# [细分领域专家 - 校园行政智能体]:
# 封装教务办事、缴费、奖学金申请等高严谨度事务逻辑。
# 对接 RAG 引擎以确切的官方制度文档指导用户办理流程。
class AdminAgent:
    """
    Administrative Agent
    Handles scholarship, leave applications, expense reimbursement,
    and administrative procedures
    """

    def __init__(self, db: AsyncSession, redis: RedisClient, rag_service: RAGService):
        self.db = db
        self.redis = redis
        self.rag_service = rag_service
```

```python
    def _identify_admin_domain(self, task_type: str, query: str) -> str:
        """基于加权关键词评分的领域识别算法"""
        query_lower = query.lower()

        # 任务类型直接映射
        task_domain_mapping = {
            "scholarship_info": "scholarship_info",
            "leave_application": "leave_application",
            "expense_reimbursement": "expense_reimbursement",
            "certificate_services": "certificate_services",
            "administrative_procedures": "administrative_procedures"
        }
        if task_type in task_domain_mapping:
            return task_domain_mapping[task_type]

        # 加权关键词评分
        domain_scores = {}
        for domain, config in self.admin_domains.items():
            score = sum(1 for keyword in config["keywords"]
                       if keyword in query_lower)
            # 紧急关键词加权
            if domain == "scholarship_info" and any(
                word in query_lower for word in ["急需", "紧急", "截止"]
            ):
                score += 2
            domain_scores[domain] = score

        if domain_scores:
            return max(domain_scores, key=domain_scores.get)

        return "administrative_procedures"  # 默认兜底
```

**技术要点讲解**：
- **加权评分算法**：对每个领域的命中关键词计数，紧急词额外加权重 → 简单但有效
- **兜底策略**：最通用的 "administrative_procedures" 作为默认 → 保证总有一个响应

```python
            search_results = await self.rag_service.hybrid_search(

                category="administrative",
                top_k=6
            )

            # 获取当前政策（Redis缓存，1天TTL）
            current_policies = await self._get_current_policies(domain)
            if current_policies:
                search_results["current_policies"] = current_policies

            # 获取相关截止时间
            deadline_info = self._get_relevant_deadlines(domain, query)
            if deadline_info:
                search_results["deadline_info"] = deadline_info

            return search_results

        except Exception as e:
            print(f"❌ Administrative knowledge search failed: {str(e)}")
            return {"semantic_results": [], "faq_results": [], "total_results": 0}
```

**技术要点讲解**：

### 4.2 多智能体调度引擎 (`backend/services/agent_orchestrator.py`)

```python
# [MAS 多智能体中枢引擎]:
# 系统架构的核心大脑。在此拦截用户输入，基于自然语言与特征词库
# 进行意图识别(Intent Analysis)和任务路由分发，实现高内聚、低耦合
# 的业务领域分离。
class AgentOrchestrator:
    def __init__(self, db: AsyncSession, redis: RedisClient):
        self.agents = {
            "navigation": NavigationAgent(db, redis),
            "academic": AcademicAgent(db, redis, self.rag_service),
            "life": LifeServiceAgent(db, redis, self.rag_service),
            "admin": AdminAgent(db, redis, self.rag_service),
            "quality": QualityAgent(db, redis)
        }

    async def process_message(self, conversation_id, user_message, user_context):
        """
        系统核心调度入口。
        1. 先根据特征词推测意图 → 决定由哪个Agent接管
        2. 行政关键词正则：奖学金|助学金|请假|报销|证明|手续|审批|缴费|费用|财务|发票
        """
        # Step 1: 意图分析
        navigation_result = await self._route_request(...)

        # Step 2: 确定性路由兜底（防止AI路由错误）
        # 如果 AI 返回 navigation/general，但用户明显问行政事务
        q = user_message.lower()
        admin_kw = r"奖学金|助学金|请假|报销|证明|手续|审批|缴费|费用|财务|发票"
        if re.search(admin_kw, q, re.IGNORECASE):
            primary_agent = "admin"  # 强制路由到行政智能体

        # Step 3: 执行主智能体
        primary_response = await self._execute_primary_task(
            conversation_id, primary_agent, task_type, query, context
        )

        # Step 4: 质量验证
        if primary_response.get("requires_verification"):
            verified_response = await self._verify_response(
                conversation_id, primary_response, user_message
            )

        # Step 5: 跨智能体协作
        if primary_response.get("requires_collaboration"):
            collaborative_response = await self._handle_collaboration(...)
```

**技术要点讲解**：
- **双层路由**：AI意图分析 + 正则强制路由 → AI可能出错，正则兜底保证准确
- **缓存防击穿**：MD5(query + user_id) → Redis → 命中直接返回，避免重复调用大模型
- **完整记录**：每次交互记录到 AgentInteraction 表 → 可追踪、可分析、可优化

```python
    # [高并发缓存防击穿拦截策略]:
    # 1. 计算用户输入特征与身份的 MD5 校验和，查询 Redis 连接池。
    # 2. 若命中重复热点查询则直接返回拦截，大幅降低并发状态下
    #    底层大模型 API 与关系型数据库的查询压力。
    async def _route_request(self, conversation_id, message, context):
        # 计算查询哈希
        query_hash = hashlib.md5(
            f"{message}:{context.get('user_id')}".encode()
        ).hexdigest()

        # Redis 缓存检查
        cached_result = await self.redis.get_cached_agent_response(
            "navigation", query_hash
        )
        if cached_result:
            return cached_result  # 缓存命中，直接返回

        # 执行意图分析
        result = await self.agents["navigation"].analyze_intent(message, context)

        # 缓存结果（30分钟）
        await self.redis.cache_agent_response(
            "navigation", query_hash, result, expire=1800
        )
        return result
```

**防幻觉机制的核心逻辑**：

```python
async def _detect_hallucinations(self, content, sources):
    """检测响应中潜在的幻觉"""
    hallucination_indicators = []
    risk_score = 0.0

    # ① 模式匹配检测
    for pattern in self.hallucination_patterns:
        matches = re.findall(pattern, content)
        if matches:
            hallucination_indicators.append({
                "type": "pattern_match",
                "pattern": pattern,
                "severity": "medium"
            })
            risk_score += 0.1 * len(matches)

    # ② 声明验证：检查内容中的具体声明是否有来源支撑
    specific_claims = self._extract_specific_claims(content)
    for claim in specific_claims:
        if not self._is_claim_supported(claim, sources):
            hallucination_indicators.append({
                "type": "unsupported_claim",
                "claim": claim,
                "severity": "high"  # 无源声明 → 高风险
            })
            risk_score += 0.3  # 每条无源声明加0.3分

    # ③ 风险分级
    if risk_score >= 0.7:   return "high"
    elif risk_score >= 0.4: return "medium"
    elif risk_score >= 0.1: return "low"
    else:                   return "minimal"
```

## 7. 技术报告

### 7.1 系统架构设计

本项目采用 **多智能体系统（MAS）+ 检索增强生成（RAG）** 的混合架构，核心设计思想为：

> "开发者绘制系统蓝图与骨架，AI负责在限定的局部函数内填补砖瓦"
> —— 引自 AI_Assisted_Coding_Log.md

#### 7.1.1 为什么选择 MAS？

| 单体AI方案 | MAS方案（本项目） |
|-----------|-----------------|
| 一个Prompt处理所有问题 | 5个专业Agent各司其职 |
| 意图混乱，回复不聚焦 | NavigationAgent精准路由 |
| 学术、生活、行政混在一起 | 领域隔离，高内聚低耦合 |
| 无法验证回复质量 | QualityAgent独立质量审查 |
| 无法发现需要协作的复杂问题 | Agent间可请求协作 |

#### 7.1.2 为什么选择 RAG？

| 纯大模型方案 | RAG增强方案（本项目） |
|-------------|---------------------|
| 幻觉率高，可能编造政策 | FAQ直接命中不走大模型 |
| 无法引用具体的校园制度 | 每个回复附知识库来源 |
| 政策更新需重新训练 | 更新知识库即可生效 |
| 回答通用化，不贴近学校 | 基于HBUE实际制度回答 |

### 7.2 关键技术挑战与解决方案

#### 挑战一：行政事务的高严谨性要求

**问题**：行政事务涉及奖学金金额、截止日期、审批流程等精确信息，AI幻觉会导致严重后果。

**解决方案**：
1. **三层防控体系**：FAQ直接返回 → RAG上下文约束 → QualityAgent后置验证
2. **代码级熔断**：知识库无命中时不调用AI，直接返回"建议联系相关部门"
3. **行政领域特殊验证规则**：
```python
"administrative": {
    "require_sources": True,       # 强制要求来源
    "check_deadlines": True,       # 验证截止日期
    "verify_procedures": True,     # 验证办理流程
    "validate_contact_info": True  # 验证联系方式
}
```

#### 挑战二：用户意图的模糊性

**问题**：用户可能问"奖学金"，实际可能是想申请、想查询条件、想了解截止日期。

**解决方案**：
1. **NavigationAgent 双层分类**：先确定大领域（admin），再细分任务类型（scholarship_info）
2. **加权关键词评分**：紧急词加权 → "急需奖学金"比"奖学金"优先级更高
3. **建议追问**：置信度不足时生成追问选项

#### 挑战三：高并发下的性能

**问题**：大量学生同时查询可能导致数据库和大模型API过载。

**解决方案**：
1. **MD5哈希缓存**：相同问题+相同用户 → Redis直接返回，零计算
2. **多级缓存体系**：本地缓存 → Redis → 数据库 → DeepSeek API
3. **异步架构**：FastAPI + SQLAlchemy Async + asyncio

### 7.3 创新点总结

| 创新点 | 具体实现 | 价值 |
|--------|---------|------|
| **行政事务七域模型** | 7个子领域 + 知识库 + 流程步骤 + 材料清单 | 覆盖面全、回答精准 |
| **两级降级策略** | AI优先 → 规则引擎兜底 | 高可用、不中断 |
| **代码级幻觉熔断** | 知识库无命中 → 不调AI → 兜底话术 | 杜绝胡编乱造 |
| **加权紧急度评估** | 关键词加权 + 紧急度分级 | 最重要的事务优先 |
| **跨Agent协作** | 文件验证/截止检查/资格评估 | 处理复杂跨领域问题 |
| **MD5缓存防击穿** | 查询哈希 + Redis拦截 | 高并发保护 |

## 9. 答辩准备（Q&A预演）

### 9.1 预期问题与回答

#### Q1：为什么选择多智能体架构，而不是一个大的 Prompt？

**回答要点**：
> 单体AI在处理多样化校园问题时存在明显的意图漂移问题。我们采用NavigationAgent做路由分发，让学术、生活、行政各有独立的Prompt和知识库，确保专家级回答质量。同时QualityAgent独立验证提供了安全网，这是单体架构无法实现的。

#### Q2：行政事务的高严谨性要求下，如何保证回复的准确性？

**回答要点**：
> 我们设计了三层防幻觉体系：
> 1. **FAQ精确匹配层**：命中标准FAQ直接返回，跳过AI生成，置信度0.98
> 2. **RAG上下文约束层**：将校园制度原文植入Prompt，强制AI基于知识库回答
> 3. **QualityAgent验证层**：检测无源声明、虚假电话号码、日期等具体幻觉信号
>
> 当三层中任何一层发现风险，系统会降低置信度或触发熔断，建议用户联系实际部门。

#### Q3：如果用户问的问题知识库里没有怎么办？

**回答要点**：
> 这是幻觉风险最高的场景。我们的处理策略是：
> 1. 如果语义搜索完全没有命中（score < 阈值），触发 **代码级熔断**
> 2. 不调用DeepSeek大模型，直接返回预定义的兜底回复
> 3. 兜底回复包含相关办公室的联系方式和办理建议
> 4. 同时记录为"未覆盖问题"，后续可以补充到知识库

#### Q4：系统如何处理多个智能体需要协作的复杂问题？

**回答要点**：
> 我们设计了三种跨智能体协作模式：
> - **文件验证**：AcademicAgent不确定材料是否齐全时，调用AdminAgent验证
> - **截止时间检查**：任何Agent检测到"紧急"关键词，请求AdminAgent查截止日期
> - **资格评估**：根据用户画像（GPA、年级等）评估是否符合申请条件
>
> 每轮协作都记录到AgentCollaboration表，方便追踪和分析。

#### Q5：为什么用SQLite而不是MySQL/PostgreSQL？

**回答要点**：
> 开发环境使用SQLite是为了零配置快速启动。但我们的架构通过SQLAlchemy ORM抽象了数据库层，生产环境只需修改.env中的DATABASE_URL即可切换为PostgreSQL，无需改动任何业务代码。这是典型的开发便利性和生产可扩展性的平衡设计。

#### Q6：智能体的关键词匹配会不会太简单了？如何应对复杂表达？

**回答要点**：
> 关键词匹配是我们的**兜底策略**，不是主要策略。优先级是：
> 1. DeepSeek AI意图分析（优先）— 能处理"我最近手头有点紧，学校有什么帮助吗"这类隐含表达
> 2. 正则强制路由（兜底）— AI可能误判时，用规则强制纠正
>
> 这个双层设计就是考虑到AI可能出错的情况，用规则作为安全网。

#### Q7：这个系统实际能处理多少并发用户？

**回答要点**：
> 我们的设计考虑了几个关键性能点：
> 1. **FastAPI异步架构** — 单个worker即可处理数百并发连接
> 2. **MD5缓存防击穿** — 相同问题直接Redis返回，不消耗AI API配额
> 3. **FAQ直接返回** — 80%的常见问题不走AI，响应时间<50ms
> 4. **水平扩展** — 通过Nginx负载均衡 + 多worker轻松横向扩展

#### Q8：你的个人主要贡献是什么？

**回答要点**：
> 我的主要贡献集中在行政事务智能体模块：
> 1. **需求分析**：调研了学校7大行政事务领域的常见问题和办理流程
> 2. **领域建模**：设计了keywords + domain + context的三层分类体系
> 3. **代码实现**：完成了AdminAgent的完整开发（700行代码）
> 4. **知识库建设**：整理了行政类FAQ和文档模板
> 5. **质量保障**：针对行政领域设计了特殊的验证规则（截止日期验证、联系方式验证等）
