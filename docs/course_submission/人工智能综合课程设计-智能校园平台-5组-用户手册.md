## 二、系统概述

### 2.1 系统简介
智能校园平台是一个基于多智能体（LLM-MAS）和检索增强生成（RAG）技术构建的智能问答平台。系统旨在解决传统校园中规章制度零散、办事流程复杂、师生咨询耗时的问题。通过整合校园内部的各类文档（学生手册、报销指南、后勤服务等），平台能够全天候为师生提供准确的资讯查询与事务办理指南。

### 2.2 系统目标用户

| 用户角色 | 使用目的 | 主要功能 |
| --- | --- | --- |
| 学生 | 快速了解选课规则、学分计算、食堂分布、奖学金申请流程、考试安排等 | 智能对话提问、历史会话查阅 |
| 教师 | 获取科研经费报销、教务流程指南 | 智能对话提问、政策查询 |
| 管理员 | 减轻重复咨询工作量；维护校园知识库的最新内容 | 上传和更新知识库文档 |

### 2.3 系统主要功能

| 功能名称 | 功能说明 |
| --- | --- |
| 多智能体流式问答 | 用户输入自然语言问题，系统自动分配专属智能体并流式输出答案，并附带引用来源 |
| 对话历史管理 | 系统自动保存用户的多轮对话记录，支持随时查看和继续追问 |
| 智能体状态展示 | 在网页端实时展示当前是哪个智能体（如学术、行政、生活等）正在思考及协作流转过程 |
| 本地知识库融合 | 将长篇的校园PDF、Word文档转化为向量存储，解决大模型在特定领域数据的幻觉问题 |

## 六、系统使用说明

### 6.1 功能一：智能校园问答
- **功能简介**：用户可通过自然语言提问获取校园相关资讯，系统通过多智能体协同作答。
- **适用对象**：所有访问网站的用户。
- **输入内容**：自然语言问题（如“图书馆周末几点关门？”）。
- **操作步骤**：
  1. 打开平台首页（默认 http://localhost:3000）。
  2. 在底部输入框输入问题，如：“大二下学期需要修满多少学分？”或“图书馆周末几点关门？”。
  3. 按回车键或点击右侧发送按钮。
- **输出结果**：页面首先展示智能体的中间思考状态，随后以打字机流式效果输出包含引用来源的答案。
- **注意事项**：请耐心等待 3-5 秒，复杂问题需要多个智能体联动检索知识库。

### 6.2 功能二：对话历史管理
- **功能简介**：保存用户的多轮对话记录，支持新建、切换或删除会话。
- **适用对象**：所有用户。
- **操作步骤**：
  1. 在页面左侧（PC端）或点击菜单栏展开左侧抽屉（移动端）即可看到历史列表。
  2. 点击顶部“+ 新建对话”可重置记忆并开启新话题。
  3. 点击任意历史记录可恢复先前的聊天内容。
- **输出结果**：主聊天区域刷新为对应的历史记录内容。

### 6.3 功能三：智能体状态看板
- **功能简介**：实时展示当前是哪个智能体在思考以及协作流转过程。
- **适用对象**：所有用户。
- **操作步骤**：无需额外操作，发送问题后页面会自动显示状态栏（如“导航智能体正在分析...”）。
- **输出结果**：动态UI组件更新。

### 6.4 典型使用流程
1. **系统登录与主界面初始化**：用户通过现代浏览器（建议 Chrome 或 Edge）访问系统的公网/局域网入口（默认 `http://localhost:3000`）。系统会自动建立一个新的匿名会话（Session），并在主界面的对话框中显示欢迎引导语。
2. **侧边栏功能选择与会话管理**：用户可以在屏幕左侧的侧边栏（Mobile端为抽屉式菜单）浏览自己近期的所有咨询记录。点击任一历史记录，系统将从后端 Redis 缓存中迅速拉取当时的对话上下文，并在主屏幕上无缝恢复。
3. **发起提问与智能体交互**：用户在底部的输入框中以自然语言描述自己的问题（例如：“我是今年刚入职的新教师，请问科研经费报销需要走哪些审批流程？”），然后点击发送。
4. **多智能体协作状态观测**：发送后，用户无需枯燥等待。界面上方会动态闪烁状态标签，例如“导航智能体正在识别意图”、“行政智能体正在查阅报销指南”、“审核智能体正在检查违规词汇”。
5. **获取智能解答与参考溯源**：流式打字机效果结束后，用户可以获得条理清晰的步骤解答。在回答的末尾，系统会折叠显示被引用的校园规定原文档出处（如《湖北经济学院差旅费报销管理办法》第X页），用户可点击以核对原文，确保政务信息的绝对严谨与准确。

### 6.5 高效提问与 Prompt 最佳实践指南（重要）
为了让底层的大模型与 RAG 检索系统更好地为您服务，我们强烈建议用户在提问时遵循以下“最佳实践”（Best Practices）：

1. **提供充分的背景信息**：不要只问“怎么转专业？”。不同的年级、不同的原始专业面临的规定截然不同。建议改为：“**我是大一下学期的新生，目前在工商管理专业，如果在期末考试中没有挂科且绩点排名前10%，想转入计算机科学与技术专业，请问需要满足什么条件？**” 这样系统能精准匹配《本专科学生转专业管理办法》中的特定条款。
2. **一个问题包含一个明确诉求**：虽然多智能体系统具备一定的复杂任务拆解能力，但将复合问题拆分为单点提问能显著提高检索准确率。避免提问：“学校的食堂在哪？顺便告诉我报销要盖几个章？” 建议分为两次单独提问。
3. **避免过度口语化或指代不明**：RAG 向量检索对具体的实体名词更为敏感。尽量使用校园官方术语。例如，将“那个拿奖学金的表格在哪里”替换为“**《国家奖学金申请审批表》在哪个部门下载**”。
4. **善用连续追问能力**：本系统具备强大的上下文记忆能力。如果您对系统的第一次回答不够满意，不需要重新复述整个问题。您可以直接追加提问：“**如果是三等奖学金呢？**” 或者 “**请把上面你说的第二步详细展开讲讲。**” 系统会自动结合前文为您继续解答。

### 6.7 移动端与多端适配使用指南
本项目前端基于 Tailwind CSS 实现了深度的响应式适配（Responsive Design），为了让全校师生在任何设备上都能获得无缝的使用体验，本节特此说明移动端设备（如手机、平板）的使用差异与建议：
1. **移动端布局自适应**：当用户使用手机浏览器（如 Safari, Chrome Mobile, 微信内置浏览器）访问平台时，PC端宽屏左侧的长条形历史会话列表将自动折叠。界面会腾出所有屏幕空间给核心的“流式对话区域”。
2. **手势与侧边栏唤出**：在手机端，用户若想查看“历史对话”，只需点击屏幕左上角的“汉堡菜单（Hamburger Menu）”图标，即可通过流畅的滑动抽屉（Drawer）动画唤出历史记录列表。
3. **输入体验优化**：移动端输入框专门为触屏键盘进行了高度优化。当手指点击输入框时，键盘弹起不会遮挡上方正在流式输出的文字；在网络环境不佳（如校园网死角）或断网时，输入框右侧的发送图标会自动变为灰色“不可点击”状态，直观提醒用户检查网络连接。
4. **长按交互支持**：移动端完全保留了对复杂排版（如代码块、Markdown 表格、加粗引用）的渲染能力。用户可以通过长按智能体输出的文本内容进行快速复制或全选分享至微信/QQ。

**【场景一：大一新生入学的“校园百事通”】**
- **用户画像**：刚刚踏入校园，对周围环境、规章制度一无所知，经常在班级群里反复询问辅导员同样的基础问题。
- **演练路径**：
  1. 新生登录平台，提问：“**马上要军训了，请问必须购买规定的军训服吗？如果有特殊疾病不能参加怎么办？**”
  2. 系统的“导航智能体”敏锐捕捉到“军训”、“疾病”关键词，将其路由至“生活服务智能体”。
  3. 生活智能体在知识库中匹配到《新生入学指南》与《军训管理规定》。
  4. 系统流式输出：“根据《新生入学指南》规定，军训服为统一采购... 若因特殊疾病（如心脏病、严重哮喘等）无法参加，需携带县级以上三甲医院出具的诊断证明，在开学第一周内向所在学院辅导员提交《军训免修申请表》...”
  5. 辅导员的重复劳动被彻底解放。

**【场景二：临近毕业的老生的“教务规划师”】**
- **用户画像**：大四学生，面临学分清查、毕业论文开题、考研保研等一系列复杂而严谨的教务节点。
- **演练路径**：
  1. 学生登录平台，提问：“**我是一网专业的大四学生，之前大二有一门选修课挂科了，后来重修通过，这会影响我的推免保研资格吗？**”
  2. 该问题极为严谨，触发“学术教务智能体”。智能体检索《推荐优秀应届本科毕业生免试攻读硕士学位研究生实施办法》。
  3. 系统回答：“您好！根据学校最新的推免办法第X条规定：凡在校期间有课程不及格记录（即使后续重修通过），原则上将失去本校推免资格。但如果您在学术顶级期刊上发表过第一作者论文，或在A类国家级竞赛中获奖，可通过‘特殊学术专长’通道进行破格申请。详情请查阅...”
  4. 这种精确到特定条件的回答，避免了学生因为信息差而错过关键机会。

**【场景三：青年教师的“行政办事小助手”】**
- **用户画像**：刚入职的青年教师或行政人员，需要频繁与人事处、科研处、财务处打交道。
- **演练路径**：
  1. 教师提问：“**我上周去北京参加了一个人工智能学术会议，住宿费超标了 100 块钱，请问这部分超标费用科研经费能报销吗？需要附带什么说明？**”
  2. 任务路由至“行政事务智能体”，在后台比对《教职工差旅费管理办法》与《科研项目经费使用细则》。
  3. 系统回答明确指出：“根据财务处规定，差旅住宿费实行限额凭据报销。超出限额标准的 100 元部分将**不予报销**，只能由您个人承担。科研经费不能用于填补超标的住宿差额。报销时只需提交发票和参会通知，系统会自动按最高限额为您结算。”

## 七、验收演示操作指引

### 7.1 现场演示准备
请在期末软件验收与答辩评审前，严格检查并准备以下内容：
- **硬件与网络**：确保演示使用的笔记本电脑已充满电，并连接至稳定高速的校园网或手机热点。由于系统高度依赖公网调用大语言模型 API，网络延迟直接决定了演示成败。
- **服务预热**：在验收前 30 分钟，确保 Redis 进程、FastAPI 后端、React 前端均已全部启动。并在浏览器中自己进行 2-3 轮提问，以唤醒大模型的冷启动和建立数据库连接池。
- **配置文件复查**：检查 `.env` 文件中的 `DEEPSEEK_API_KEY` 是否有效且有充足余额。检查 `DATABASE_URL` 路径是否正确。
- **应急预案（Plan B）**：强烈建议在本地准备一份系统功能的完整高清录屏视频（包括提问、知识库搜索、多智能体状态跳动）。一旦现场遇到极其恶劣的网络风暴或大模型服务商宕机，立刻切入视频演示，并向评委解释不可抗力原因。

### 7.2 建议演示流程（剧本式大纲）
在短短 5-10 分钟的答辩演示环节中，建议主讲人按照以下“剧本大纲”进行紧凑而富有逻辑的展示：

1. **一分钟开场（系统定位与痛点）**：
   - “各位评委老师好，我是第5组的演示人。今天我们展示的是《智能校园平台》。大家知道，每到开学季教务处老师总会被‘怎么选课’、‘怎么转专业’问得焦头烂额。今天我们带来的系统就是为了彻底消灭这种重复劳动。”
2. **两分钟核心亮点演示（多智能体与 RAG）**：
   - 现场打开系统首页，模拟一个非常刁钻的具体问题输入（例如：“我是留学生，我丢了校园卡去哪里补办，需要多少钱？”）。

## 八、常见问题与高级排错说明

### 8.1 基础错误排查表

| 问题现象 | 可能原因 | 解决办法 |
| --- | --- | --- |
| 页面无法访问 (白屏或 404) | 前端未启动或端口(3000)被占用 | 检查 Node 终端状态。若报错端口冲突，请在任务管理器中杀死占用 3000 端口的进程，或在 `package.json` 中修改启动端口后重启 `npm start`。 |
| 后端报错或接口持续返回 500 | Redis 缓存服务未启动 | 检查操作系统的 Redis 进程是否在后台运行。Windows 下可查看任务管理器的 `redis-server.exe`。 |
| 发送问题后控制台报错 Auth 失败 | 大模型 API Key 失效、拼写错误或欠费 | 检查后端根目录 `.env` 文件。确认 `DEEPSEEK_API_KEY` 是否以 `sk-` 开头。登录大模型服务商控制台查看账户余额是否大于 0。 |
| 无法回答校园问题 / 提示查无此文 | FAISS 向量库未初始化或未成功构建 | 在终端进入 `backend` 目录，重新运行 `python scripts/import_hbue_knowledge.py`，观察控制台是否有“成功将 X 块文本灌入向量库”的提示。 |
| 前端 Console 疯狂提示 CORS (跨域错误) | 后端异常挂掉或未配置正确的白名单 | 确认 FastAPI 已经正常在 8000 端口启动。并检查 `core/config.py` 中的 `BACKEND_CORS_ORIGINS` 是否包含了前端当前的实际访问地址（如 `http://localhost:3000`）。 |
| 模型加载失败 / 连接超时 | 权重路径错误或本机网络代理配置不当 | 本项目使用云端大模型 API，如果您的电脑开启了不兼容的 VPN/Clash 全局代理，可能导致 Python 无法发起外部 HTTPS 请求。请尝试关闭 VPN 后重启后端服务。 |
| SQLite 提示 Database is Locked | 高并发导致的本地文件锁 | 如果多人同时进行高频提问，SQLite 文件容易被锁。请运行根目录的 `clear_all_cache.ps1` 后重启，或升级至 PostgreSQL。 |
| 接口无返回 / 前端一直在转圈 | 后端线程发生死锁或进入死循环 | 查看后端的控制台运行日志，若发现一直停留在某一个 Agent 的执行步骤不往下走，直接按 `Ctrl+C` 强杀终端并重新启动。 |

### 8.2 高级故障排查与日志分析指南
对于接手本项目的开发和维护人员，遇到疑难杂症时，应当熟练掌握通过日志进行排查的技巧：

1. **追踪 SSE 断流问题**：
   - 现象：前端打字机效果突然卡住，但并未报错。
   - 排查：打开浏览器开发者工具 (F12) -> Network -> 选择 Fetch/XHR 标签 -> 找到 `/api/chat/stream` 的长连接请求。查看 `EventStream` 选项卡，观察接收到的最后一个 JSON 块是什么。通常是因为大模型生成的文本突然包含了非法的 Unicode 字符导致后端序列化失败断开。
2. **分析 CrewAI 智能体死循环**：
   - 现象：后台日志疯狂输出，同一个智能体反复调用同一个检索工具却找不到结果，导致 Token 消耗极快（即 Reflection 死循环）。
   - 排查：在 `.env` 中设置 `DEBUG=true`。观察控制台打印的 LangChain 详细执行链路。如果智能体陷入了“由于搜索不到知识库 -> 决定更换关键词再搜索 -> 依旧搜索不到”的死循环，需要通过干预 Prompt，强制要求智能体在“尝试三次失败后立即放弃并向用户致歉”。
3. **定位 Redis 造成的幽灵上下文**：
   - 现象：用户明明已经点击了“新对话”，但系统依然在回答上一个对话的问题。
   - 排查：进入 Redis 命令行工具（`redis-cli`），输入 `keys *`。查看当前存储的 Session ID 是否被正确清理。这通常是因为前端虽然清空了 UI，但没有将重置指令成功发送给后端，导致后端依然从 Redis 提取了历史 Context 注入到大模型中。

## 九、安全、隐私与使用限制

### 9.1 数据安全说明
- **数据保存位置**：系统不会收集用户的个人身份信息。对话内容缓存在本地的 Redis 数据库中，知识库向量存储在本地磁盘。
- **数据访问权限**：仅本地部署用户可访问。
- **是否进行脱敏处理**：知识库数据均为公开校园文件，无敏感信息。
- **是否支持日志审计**：后端控制台可查看完整的请求与代理流转日志。

### 9.2 隐私合规说明
- 知识库中使用的测试数据（如《学生手册》、《报销指南》）均来源于高校官方公开发布的文件，不包含师生的学号、密码、联系方式等任何敏感信息，完全符合合规性要求。

### 9.3 使用限制说明
- **无网络环境不可用**：系统强烈依赖公网，若验收现场断网，只能使用准备好的录屏视频进行展示。
- **实时数据受限**：平台目前为纯文档检索（RAG），尚未接入实时动态的教务 API。因此无法回答类似“这节课在哪个教室”、“我的期末成绩是多少”等个人动态信息。
- **仅支持现代浏览器**：建议使用 Chrome/Edge 最新版。
- **单用户测试模式**：本地演示主要针对单机测试，未配置企业级多用户并发隔离机制。
- **特定场景效果下降**：对于模糊或高度主观的问题（如“哪个食堂最好吃”），回答可能偏通用。

## 十、系统维护说明

### 10.1 日常维护与数据监控
为了保证智能校园平台长期的生命力，运维人员需要进行以下常规维保动作：
- **密钥与成本监控**：需定期登录大模型服务商（如 DeepSeek）控制台，检查 API 的调用量消耗图表。一旦发现调用激增（可能遭遇恶意刷接口攻击），应立即在后端网关处封禁恶意 IP，并重置 `DEEPSEEK_API_KEY`。
- **日志轮转清理**：平台产生的后台日志、访问日志会逐渐侵占磁盘空间。建议配置 Linux 自带的 `logrotate` 服务，对 `backend/logs/` 下的文件进行每天切分，并仅保留最近 30 天的日志。

### 10.2 后续升级与架构扩展
本平台在设计之初就考虑了极强的可扩展性。随着学校业务的发展，技术团队可以沿着以下路径进行系统演进：
- **打通实时教务 API (Tool Calling 升级)**：目前的 RAG 技术只能回答静态规定。未来可以在代码中引入 LangChain 的 `Tools` 机制，将学校真实的“教务排课接口”、“一卡通余额查询接口”封装为函数。当学生提问“帮我查一下校园卡还有多少钱”时，智能体能够自动触发调用对应的 API，将查询结果反馈给用户。
- **高可用分布式演进**：当平台开放给全校两万名师生同时使用时，单机部署的 FastAPI 将无法承受。后端架构支持轻松使用 Docker 构建镜像，并部署至 Kubernetes (K8s) 集群。通过 Nginx 或 Traefik 进行四层/七层负载均衡，以支撑万级高并发访问。
- **多模态与视觉识别**：未来可更换为支持多模态输入的大模型（如 GPT-4o 或 Claude 3）。从而允许学生直接拍摄一张“损坏的宿舍水管”图片或一张“看不懂的选课网页截图”发送给平台，生活智能体通过视觉理解能力给出更直观的维修建议。

### 10.3 管理员进阶手册：知识库动态维护与调优
本系统的“聪明程度”与“少犯错”，完全取决于底层 FAISS 向量知识库的质量。因此，管理员的“喂饭”技巧至关重要：

1. **原文档的规范化预处理（极度重要）**：
   大模型并不是什么格式的文档都能完美阅读。在向 `data/` 目录上传新的校园规章 PDF 或 Word 之前，管理员必须人工进行以下预处理：
   - **剔除视觉干扰**：删除 PDF 中每一页的页眉、页脚、复杂的背景水印，这些无意义字符被切分进向量块后会严重干扰大模型的注意力。
   - **表格 Markdown 化**：如果原始文件中包含复杂的“学分替换标准对照表”、“食堂价格表”，请务必利用工具将其转化为标准的 Markdown 表格格式后再上传。否则大模型在读取打散的表格文本时必定发生串行错位，引发回答灾难。
2. **重构向量库与分块参数调优 (Chunking Optimization)**：
   每次文档更新后，必须重新运行初始化脚本 `import_hbue_knowledge.py` 覆盖旧的 FAISS 索引。在处理过程中，如果发现大模型在回答时“经常断章取义，只回答了一半”，请联系开发人员修改后端 RAG 服务中的 `chunk_size`（分块大小）和 `chunk_overlap`（重叠大小）参数。通常将重叠值调大，可以有效防止一条完整的规章制度被强行劈成不连贯的两段。
3. **建立定期“纠偏”机制**：
   管理员应定期抽查用户的历史提问记录。如果发现大量用户提问“体育馆怎么借拍子”，而系统的回答总是答非所问，这说明知识库中缺乏此类信息。管理员应迅速整理一份针对性的 QA 问答文档录入系统中，实现“打补丁式”的敏捷优化。

### 11.4 系统底层交互原理解析（面向高阶用户与评审专家）
为了帮助计算机专业的测试人员、评委以及对技术原理感兴趣的极客用户更深刻地理解本系统的“黑盒”运作机制，本节以“用户提问”为起点，梳理后台系统内部各组件相互调用的全生命周期（Lifecycle）：

1. **Step 1：长连接建立与降级验证（SSE Initialization）**
   - 传统 Web 系统是“一问一答”的同步请求，而大模型的思考时间极长。因此，当用户点击“发送”按钮时，浏览器实际上发起了一个 `text/event-stream` 请求。FastAPI 后端在接到请求的最初几毫秒内，首先进行极速的 Redis 健康检查，若 Redis 宕机，则系统内部立即启动“无缓存降级模式”，随后在控制台输出 `[INFO] SSE connection established.`。
2. **Step 2：导航分发与上下文注水（Router & Context Injection）**
   - 后端通过 `Session_ID` 提取出用户前10轮的提问记录。这些记录连同刚刚输入的新问题一起被打包。CrewAI 框架首先唤醒“导航智能体”。由于导航智能体内部被植入了 Zero-Shot 与 Few-Shot Prompt（例如：“如果问题包含‘食堂’、‘浴室’，必须将其分类至类别 [LIFE]”），它能以极快的速度（通常只调用轻量级模型或走匹配规则）决定下发路径。
3. **Step 3：知识库向量召回与重排序（Vector Retrieval & Reranking）**
   - 假设请求被分配至“学术智能体”。此时大模型实际上处于静默状态，系统先进入 RAG 链路。Python 会调用轻量的 Embedding 算法（如 `all-MiniLM-L6-v2`），将用户的自然语言（如“挂科怎么办”）转化为一个含有几百个维度的浮点数数组（即向量）。
   - 这串数组随后被扔进 FAISS 引擎的浩瀚星海中。FAISS 使用余弦相似度（Cosine Similarity）或是 L2 距离，在毫秒级内从几万块切片好的校园手册切片中，捞出距离最近的 Top-K 个文本段落。为防止捞出的段落前言不搭后语，系统还会带出这几个段落的原始文件名（如 `教务处规定.pdf`）与上下文边界。
4. **Step 4：思维链拼接与大模型推理（CoT & LLM Generation）**
   - 此时，组装流水线开始运作。后台自动生成类似这样的终极隐秘 Prompt：*“你是一个严谨的高校教务老师。请根据以下提供的【参考资料：XXX】，回答学生的提问：【挂科怎么办】。要求：1. 不允许回答资料外的内容。2. 必须在结尾标明你参考的是哪份资料。”*
   - 这段组合好的超长文本被发送至外部的 DeepSeek V3 API。大模型开始运用其千亿参数的算力，理解资料并逐个生成 Token（单字）。
5. **Step 5：质量拦截与流式冲刷（Quality Review & Streaming）**
   - 如果本系统未开启“审核智能体”，大模型生成的 Token 会被 FastAPI 的生成器（Generator）通过 `yield` 命令立即吐向前端。
   - 如果开启了质量审核智能体，系统会在内部将生成的初稿做一次正则校验，检测是否存在不文明用语或明显的幻觉信息（Hallucination）。一旦通过，这一大段带有 Markdown 格式标记的字符串便如流水一般，通过建立好的 SSE 通道，冲刷至用户的屏幕上，最终呈现出充满科技感的打字机效果。
